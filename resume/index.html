소 개

업무에서 저는 대부분 스스로 했던 것 같습니다. 일의 긴급성에 따라서는 도움을 요청했고, 여러 방안을 찾았을 때 어떤 선택이 최선인지에 대한 자문은 구했지만, 대부분 스스로 해결하고 있습니다. 그런 식으로 일을 해왔습니다.

처음, 직군 변경의 이유로는 서버 엔지니어 2년 동안 단순히 메뉴얼에 있는 명령어들을 타이핑만 한다는 느낌에 이직을 결심하게 되었고, 서버 직군을 하면서 Spring기반의 웹 개발을 경험했지만 이미 정해진 패턴에 IF~ELSE만 붙이는 웹 개발은 미래가 없다고 생각이 들었습니다. 그래서 생각하는 개발을 하고자 휴식기간동안 C++를 배우면서 FTP를 만들었습니다. 처음에 아무것도 없는 빈 파일에서 main 함수를 시작할 때 첫 타이핑을 하기까지 매우 오래걸렸던 기억이 있습니다. 어떻게 각 모듈을 분리해야하는지 배운적도, 해본적도 없어서 실제 기능을 짜기 전부터 수차례 다시 구조를 만들기를 반복했었습니다.  RFC에도 정의되지 않는 내용들은 직접 패킷을 분석하면서 개발을 진행하였고 이런 생각하는 개발이 적성이 맞다고 더욱 확신했습니다.

그러던 중 C++ 개발을 하는 회사에 이직하였고, 퇴직을 한지 1년 정도 흐른 후 지금의 대표님과 일을 하면서 이직 준비를 해도 된다는 대화를 하고 다시 첫 회사로 재이직하게 되었습니다.

계획대로 C++에 대해 더 공부하고자 했습니다. 이직 준비를 하면서 떠나도 좋다라는 말씀을 들었지만 맡은 업무 수행에 있어서 기본은 하자는 생각에 javascript를 배우게 되었습니다. 이전 웹 개발 경험에서 javascript는 문자열 검사나 select 박스정도 다루는 언어라고 생각했었습니다. 그러나 깊게 배워보니 C/JAVA에 비해 강제성은 없어 다루기는 까다롭다면 까다로웠지만 그만큼의 자유를 허락하는 JS의 매력에 굉장히 빠져들었고, 현재 트랜드도 더 이상 브라우저에 의존하지 않는 점등의 이유로 JS, 프론트엔드 개발자로 이직을 결심하게 되었습니다.

JS의 경험은 대표적으로 github에 있는 canvas기반의 차트입니다. 처음에는 d3.js와 같은 시각화 툴을 사용하려고 했지만 “단순히 써봤다”보다는 그것을 만들어보면 어떨까 생각하여 HTML5의 canvas태그를 공부하였고, Chart.js 소스를 분석하여 차트의 실행 부분 정도, 수치 계산 등의 힌트는 얻되 Copy & Paste는 하지 않으면서 실제 기능과 똑같이 구현하면서 JS를 익혔습니다. 그 후 단 하나의 .js파일에서 다른 언어처럼 파일을 나누어야한다는 생각에 모듈 시스템, webpack을 적용해나가면서 번역을 해보고 있습니다.
